# Plan: Fix `Start Listening` decode failure

## Main issue
`Start Listening` is not broken at the microphone level. It captures audio and logs activity, but the receiver only decodes the **last 3 seconds** of audio while the transmitted frame is **longer than 3 seconds**.

With the current protocol:
- `BIT_DURATION = 100ms`
- `REPEAT_BITS = 2`
- every payload bit is duplicated
- the message is framed with `START_MARKER` and `END_MARKER`

Frame time is therefore:
- `((START + payload + END) * repeatBits * bitDuration)`
- even an empty payload is about `4.8s`
- any real message is longer than the current `3s` receive buffer

That means even a short message can take longer than the receive window, so `Decoder.decodeBits()` never sees both markers in the same bitstream and returns an empty string.

## Fix plan
1. **Patch the receiver scope**
   - Files: `Receiver.kt`, `MainViewModel.kt`, `Constants.kt`.
   - Replace the hardcoded 3-second decode slice with a buffer that can hold a complete frame for the app’s supported message size.
   - Keep sliding-window scanning, but decode from the full retained buffer instead of only the newest 3 seconds.
   - Preserve the tail only after a successful decode so the same message is not re-read.

2. **Make the decoder tolerant of full-frame capture**
   - File: `Decoder.kt` if needed.
   - Keep START/END marker parsing.
   - Add logs when markers are missing so failure mode is explicit.
   - Do not change the bit protocol.

3. **Add hard diagnostic logs for receive scope**
   - Log rolling buffer duration in seconds.
   - Log the active decode window size.
   - Log whether START/END markers were found.
   - Log the decoded bitstream length before final text decode.
   - Log successful frame completion and why a frame was rejected.

4. **Keep the protocol intact**
   - Do not remove FSK.
   - Do not remove START/END markers.
   - Do not remove repeat-bit handling.
   - Do not remove sliding-window receive logic.
   - Do not remove cooldown or buffer trimming.
   - Do not rewrite the app structure.

5. **Validate on-device with realistic cases**
   - Empty message.
   - Short message like `HELLO`.
   - Longer text that previously failed.
   - Repeated send taps.
   - Two different phones.

## Success criteria
- `Start Listening` decodes the sender message on real devices.
- Logs show the receiver retained enough audio to contain the full framed transmission.
- Exactly one message is displayed per send.
- No duplicate or partial decodes appear after the cooldown.
- Audible test sweep still works unchanged.
- The receiver log explicitly shows when a frame was rejected because the marker pair was not fully present.
